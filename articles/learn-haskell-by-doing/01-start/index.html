What this guide is about

In this project, we are going to develop a small GraphQL server with

  • a simple HTTP server
  • a code-first GraphQL schema
  • business logic with an ORM
  • an authentication and authorization system

Before starting this project, you should know about

  • HTTP servers
  • GraphQL
  • JWT

While you can probably learn about those while following this guide, I recommand building some knowledge first, so that *you focus on Haskell here.

Development environment

If you don't know how to setup a development environment for Haskell, check out my guide "Setup a Haskell development environment in 2020"

Setup the project

Let's first create our project.

stack.yaml

resolver: lts-16.26
packages: ["."]

package.yaml

name: haskell-graphql-example
version: 0.0.0
description: An example of GraphQL server in Haskell for educational content

author: Guillaume Desforges <guillaume.desforges.pro@gmail.com>
homepage: http://guillaumedesforges.github.io/articles/learn-haskell-by-doing/01-start

dependencies:
  - base

library:
  source-dirs: src/lib

executable:
  source-dirs: src/exe
  main: Main
  dependencies:
    - haskell-graphql-example

Plus, create folders ./src/lib/ and ./src/exe.

If you don't know about those files, it means you have not checked out my guide "Setup a Haskell development environment in 2020" when you should! Come back here after reading it. 😉

Now let's write our first Haskell code:

src/exe/Main.hs

module Main where

main = print "Hello world"

check that it compiles:

$ stack build

and that it runs

$ stack run
"Hello world"

Yosh, everything is set, let's talk about Haskell!

How does it work ?

Haskell applications are typically structured as follows:

  • a library component that contains all the code that we use
  • a executable component, which is a very thin wrapper over our library

In Haskell, only the library component can be imported by other components or project, which is why almost all code should fall into the ./src/lib folder. Since we have not started anything yet, this folder is empty, but don't worry: this won't be for long.

Then comes the executable components. In most applications, you will have only one executable component, hence the use in package.yaml of the top-level executable key.

This component requires two things. First, the executable.source-dirs key, which defines the folder ocntaining our executable source code. Then, the executable.main key, which defines which module we should use as our program entry point.

Module ?

In any programming language, it is important to organize the source code. In practice, we want to split the source code into "containers" from which we can pick whatever we need whenever we need it.

Imagine that you do some DIY. You would have a lot of tools and stuff from all your different needs: screwdrivers, paint brushes, glue, ... If you have a lot of these, would you rather have it all in one messy place, or make smallers compartments that correspond to the different activities, e.g. painting, scrapbooking, wood-turning, ...?

A module is the way to make "containers of pieces of code" in Haskell. Each module is a set of top-level definitions that can be imported by other modules.

Each module is characterized by a module path. Very much like Java's package notation (e.g. com.somepackage.SomeClass), Haskell uses dot notation which is Path.To.Folder.ModuleName. By convention, the module path should reflect the source file path. For instance, if I have a file in a folder ./src/lib/Foo/Bar/MyModule.hs in the library component, the module name should be Foo.Bar.MyModule.

This module name is defined explicitly at the very beginning of the file:

./src/exe/Main.hs

module Main where

Since the file is ./src/lib/Main.hs, the module name is just Main.

Entry point?

Exactly like C, Java and most programming language, we need a "main" top level definition. By defining such a "main" function to call, programers in imperative languages (like C, C++, Java, ...) can call other functions, which can in turn call other functions, etc... which imperatively define the behavior of the program. In some other language, for instance script languages like Python, there might not always be a "main" function, but rather a specific file to interpret in order to start the application.

Haskell is like C or Java, we define a top-level main element.

./src/exe/Main.hs

main = print "Hello world"

We will talk a bit more about this main element later, because it is fondamentally different in Haskell compared to other programming languages.

The GHC compiler will be told to look for a Haskell source code file, and it will automatically try to find this main element in the file. It will then read the instructions defined in it in order to produce the output executable.

Here, the main element is set to the action print "Hello world" which, surprisingly enough, prints "Hello world" to the console!

stack run is a shortcut that triggers a build of both the library and the executable components of our package, and then calls the binary executable that was created from it (if the build didn't fail).

Since we defined in our package.yaml file the executable.source-dirs: src/exe and executable.main: Main, the compiler looks for the Main module of path ./src/exe/Main.hs, looks for the element main inside of it and uses it as the entry point of our program. It finds that the main points to print "Hello world", compiles the instructions, and booya, we got ourselves a binary executable!

Setup a Haskell development environment in 2020

- Tags: software engineering, haskell, vs code, guide, beginner
> A complete guide for beginners

Let's face it, Haskell has a bad reputation. It is known as a complicated programming language, very different from the others, which has a steep learning curve and low accessibility to newcomers. As a pretty recent Haskeller, in my humble opinion, the fact that Haskell's tooling is a mess must have been one of the most frightening thing for beginners.

I remember a year ago when I tried to start my very first Haskell project, I got very confused about which way Haskell applications should be created, written, built and run. This is clearly the worst thing that can happen to any beginner in any technology: not being able to have a "Hello World" app in a few minutes.

I think Haskell is an amazing language which is worth learning. There are probably even more posts about Haskell's greatness than posts against it. But it all goes to nothing if you can't convince others to use it because the ecosystem is just not usable.

Developers now have a pretty high standard about programming ecosystem, and they are right! The Haskell community has worked hard and many improvements have landed in 2020, making it much more usable.

Are you a beginner who wants to try Haskell? Welcome, I'll help you setup your project and run your first "Hello world"!

What this guide is not about

  • This guide is not about "the best way to create a Haskell project"
  • This guide is not about "the fastest way to create a Haskell project"

The modern code editor: VS Code

In this guide, we will use VS Code as our code editor. VS Code is for me a very versatile editor, not too heavy, extensible, and can be turned into an IDE for almost any programming language thanks to the LSP protocol.

The idea behind the Language Server Protocol (LSP) is to standardize the protocol for how such servers and development tools communicate. This way, a single Language Server can be re-used in multiple development tools, which in turn can support multiple languages with minimal effort.

You can install VS Code here: https://code.visualstudio.com/

After installing VS Code, we need to install the Haskell VS Code extension. It is a very recent extension that will allow us to use HLS, the Haskell Language Server (learn more about it here).

A bit of background: for a very long time, Haskell did not have any IDE. Some people developped HIE, Haskell IDE Engine, but it was very heavy. Some other people developped GHCIDE, a lightweight IDE engine, but it lacked functionality. The two teams decided to join forces, giving birth to HLS, "one IDE to rule them all".

If you are on one of the most common OS (Windows, MacOS or Linux), the extension will automatically download the binaries of the Haskell Language Server for you. You have nothing more to do than to install the VSCode extension!

If you have a more exotic OS or installation, you might need to install HLS yourself. Fill an issue here if you have trouble doing so.

Open VS Code, create a folder for our Haskell project (e.g. /home/youruser/haskell-guide on Linux or C:\Users\youruser\haskell-guide on Windows). From this point onward, this guide will consider this folder to be the current working directory.

If you have never used VS Code, consider following its official guide.

Create a Haskell project using Stack

Understanding the tools

The tools we will be using are GHC, Cabal and Stack. It is important to understand what each one of them does.

The Glasgow Haskell Compiler, or GHC, is the most widespread and used compiler for Haskell. It is the program in charge of turning your Haskell files into binaries, much like gcc in the C ecosystem. Close to GHC is GHCi, which is a tool to run interactively Haskell in a REPL. Most Haskell tutorials will make you use GHCi to play around with Haskell, but that is not our purpose here.

Note that other compilers exist, for instance GHCJS which turns Haskell code into JavaScript.

Like gcc, it is very tedious to use GHC by hand, especially once you want to use other librairies as dependencies.

Here comes Cabal, a system for building and packaging Haskell librairies and applications. It is pretty much the standard packaging tool used in the Haskell ecosystem. Much like CMake in C, Maven in Java or npm in JavaScript, the user writes a file defining the package and the tool automate a lot of tasks.

Early note to save you time later: do not think of cabal install as npm install, they are not the same. Just do not use it, never. Not until a later time.

While Cabal could be sufficient to create a Haskell package, one would still have to manage many different things to get a program to compile. Which GHC version? How to install dependencies? How to compile everything for the right GHC version? It's all too tedious for us for now. Fortunatly, there's another too for use.

Meet the "Haskell Tool Stack", or Stack, a cross-platform program for developping Haskell projects. Stack will be the only tool we interact with. It is on top of Cabal and GHC and will automate and simplify our workflow by calling Cabal whenver it is needed, which will call GHC whenever it is needed.

If you are on NixOS, be mindful that Stack will automatically enable Nix support.

Stack in practice

First, install it following the official instructions.

Configure Stack

We now need to configure Stack for our project. This is done by creating a ./stack.yaml file:

# Use LTS of Stackage for GHC 8.4.4
# https://www.stackage.org/lts-16.26
resolver: lts-16.26

# Local packages
packages:
  - ./helloworld

Let's walk through this file, there are some really important things here.

resolver: define Stackage snapshot

The resolver attribute defines which snapshot of Stackage the Stack tool should use.

Stackage is a repository of packages. While Hackage is the official package repository where all Haskell packages are published, Stackage provides regular "package sets" (snapshots) that each corresponds to a GHC version and comprises Haskell packages that all build together. This is very handy to ensure that you don't have major compatibility problems between dependencies, without being aware of potential breaks.

It is important to understand that Haskell, while being ready for commercial use, is also a subject of research and development. This means that things can get very different between GHC version. While new functionalities are often published, things can suddendly break after an upgrade of GHC. If you are a beginner, you should most likely use a fairly recent version of GHC and stick with it for a while.

In this guide, I will use the lts-16.26 snapshot, which corresponds to the package set for GHC 8.4.4. You can find all snapshots on their website.

packages: define local packages

A project defined with Stack can hold multiple packages. A project repository holding multiple packages (librairies and programs) is often called a "monorepo".

The packages attribute should hold a list, each element representing a path to a subfolder holding a package. In this guide, we will have only one package, contained in the ./helloworld directory.

Create a package

Create the subfolder ./helloworld.

When using Stack, we can define a package using hpack. What we have to do is to simply create a package.yaml file. Stack will call hpack, which will turn it magically into a Cabal project for us!

Let's write to the ./helloworld/package.yaml file:

name: helloworld
version: 0.0.0
description: My first Haskell package

dependencies:
  - base

library:
  source-dirs: ./src/lib

executable:
  source-dirs: ./src/exe
  main: Main
  dependencies:
    - helloworld

Again, let's walk this file together!

name, version and description

Some attributes are purely descriptive. You can guess the meaning of name/version/description. There are some other attributes that you can use, for instance author, license, github, ...

dependencies

Like any ecosystem, we can make librairies and applications that rely on other libraries in order not to code everything ourselves.

This list defines the packages that our package depends on, from the selected resolver (that we have chosen before in stack.yaml). This is very much like any kind of dependency definition, same as in package.json for JavaScript or requirements.txt for Python.

Our project will not have any "external" dependency, but we still need to include the base package which is Haskell's standard library:

dependencies:
  - base

Imagine you would want to use the text package, simply set dependencies to:

dependencies:
  - base
  - text
library

A Haskell package defined with Stack can be split into components: library, executables, tests.

The library attribute will allow us to define everything about the library component of our package.

Note that only the library component can be imported when using another package as dependency. Usually, almost everything in a package is defined in the library component, and the executables and tests only make the right calls in the internal library.

Each component of the package can be stored in its own folder. Here, I specify that all the Haskell source code for the library component of our package will be in ./helloworld/src/lib using the library.source-dirs attribute:

library:
  source-dirs: ./src/lib
  ...

The path given to library.source-dirs is relative to the package.yaml file of the package.

There are many more attributes you can define, but right now this is enough.

executable

Since we want to make an executable that says "Hello world", we need to define an executable component in our package. The executable attribute defines just that!

A package can define multiple executable components, but executable is useful when there is only one executable component to define.

The executable.source-dirs attribute is the same as library.source-dirs. Here, all the source code for the executable of our package will be in ./helloworld/src/exe.

The difference is the executable.main:

executable:
  ...
  main: Main
  ...

executable.main defines which Haskell module (basically which source file) contains the main top-level definition. Basically, the entry point of our executable will be in ./helloworld/src/exe/Main.hs. Worry not about all that, I will tell you more in a moment!

What might come as unexpected is that, by default, the executable and test components do not define the library of the same package as a dependency. This has to be defined manually, for instance in executable.dependencies:

executable:
  ...
  dependencies:
    - helloworld
  ...

Let's code !

Our package has the following structure:

  • helloworld folder

    • package.yaml file
    • src folder
    • lib folder
    • exe folder

First, let's create a simple source file for our library, ./helloworld/src/lib/Helloworld.hs:

module Helloworld (message) where

message :: String
message = "Hello world"

and then define our executable source code:

module Main where

import Helloworld (message)

main = putStrLn message

Normally, you should notice that when you hover over putStrLn... documentation should be appearing! This is the work of the VS Code Haskell extension.

Sometimes the Haskell Language Server takes a bit of time to start, check the status bar at the bottom of VS Code.

We can then compile our project using:

stack build

or run the executable using

stack run

which should say, after some compilation gibberish... "Hello world" !

What now ?

Congrats, your setup is ready!

In order to close this blogpost, I want to quote Alexis King, who has written a splendid guide on Haskell (not for beginners though, hence not really a guide).

Many on the "outside" seem to view Haskellers as an elitist, mean-spirited cult, more interested in creating problems for itself than solving them.

That perception is categorically wrong.

I have never been in a community of programmers so dedicated and passionate about applying thought and rigor to building software, then going out and actually doing it. [...] One of my primary motivators for writing Haskell is that it is fun and it challenges me intellectually in ways that other languages don’t. But that challenge is not a sign of uselessness, it is a sign that Haskell is so close to letting me do the right thing, to solving the problem the right way, to letting me work without compromises. When I write in most programming languages, I must constantly accept that my program will never be robust in all the ways I want it to be, and I might as well give up before I even start. Haskell’s greatest weakness is that it tempts me to try.

Now that you have a good setup to start writing Haskell, let yourself be tempted as well. 🙂

If you are still a bit afraid of Haskell, you should start reading "Learn You a Haskell for Great Good!" which is a soft introduction for beginners.

Happy hacking!

Notes
